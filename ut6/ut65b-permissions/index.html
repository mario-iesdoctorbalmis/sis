Permisos y Seguridad de Objetos en Windows Server

Permisos

Section titled “Permisos”

¿Qué son los permisos?

Section titled “¿Qué son los permisos?”

Los permisos son un componente clave de la arquitectura de seguridad de Windows Server que puede usar para gestionar el proceso de autorizar a usuarios, grupos y equipos a acceder a objetos en una red.

Los permisos permiten al propietario de cada objeto protegido, como un archivo, objeto de Active Directory o clave de registro, controlar quién puede realizar una operación o un conjunto de operaciones en el objeto o propiedad del objeto.

Control de acceso discrecional

Section titled “Control de acceso discrecional”

Debido a que el acceso a un objeto está a discreción del propietario, el tipo de control de acceso que se utiliza en Windows Server se denomina control de acceso discrecional.

Características importantes:

  • Un propietario de un objeto siempre tiene la capacidad de leer y cambiar los permisos en el objeto
  • Los permisos se aplican a objetos protegidos, como archivos y carpetas, objetos de Active Directory, servicios u objetos de registro
  • Los permisos se pueden otorgar a un usuario, grupo o equipo

Concesión y denegación de permisos

Section titled “Concesión y denegación de permisos”

Permisos implícitos:

Cuando NO se otorga explícitamente el permiso para realizar una operación, este se deniega implícitamente.

Ejemplo: Si Alicia permite que el grupo Marketing, y solo el grupo Marketing, tenga permiso para leer su archivo, los usuarios que no son miembros del grupo Marketing tienen implícitamente denegado el acceso.

Permisos explícitos:

Los permisos también se pueden denegar explícitamente.

Ejemplo: Alicia podría no querer que Roberto pueda leer su archivo, aunque él sea miembro del grupo Marketing. Puede excluir a Roberto denegándole explícitamente el permiso para leer el archivo.

Advertencia sobre denegaciones explícitas

El uso de denegaciones explícitas, aunque posible, aumenta la complejidad de la política de autorización y puede crear errores inesperados.

Ejemplo: Es posible que desee permitir que los administradores de dominio realicen una acción pero denegar a los usuarios del dominio. Si intenta implementar esto denegando explícitamente a los usuarios del dominio, también deniega a los administradores de dominio que también son usuarios del dominio.

Aunque a veces es necesario, puede y debe evitar el uso de denegaciones explícitas en la mayoría de los casos.

ACE y DACL

Section titled “ACE y DACL”

Cada permiso que el propietario de un objeto otorga a un usuario o grupo en particular se almacena como una ACE (Access Control Entry - Entrada de control de acceso) en una Lista de control de acceso discrecional (DACL) que forma parte del descriptor de seguridad del objeto.

Descriptores de seguridad

Section titled “Descriptores de seguridad”

Cada contenedor y objeto en la red tiene un conjunto de información de control de acceso adjunta que se denomina descriptor de seguridad.

  • El descriptor de seguridad se crea automáticamente junto con el contenedor u objeto
  • El descriptor de seguridad de un objeto contiene sus permisos asociados a usuarios y grupos específicos

El conjunto completo de entradas de permisos en un descriptor de seguridad se conoce como ACL (Access Control List - Lista de control de acceso).

Propiedad (Ownership)

Section titled “Propiedad (Ownership)”

Propietario de objetos

Section titled “Propietario de objetos”

Cada objeto tiene un propietario, ya sea en un volumen NTFS o en Active Directory.

El propietario controla cómo se establecen los permisos en el objeto y a quién se otorgan los permisos.

Características importantes:

  • Por defecto, en Windows Server, el propietario es el creador del objeto
  • Si el creador de un objeto es miembro del grupo Administradores, el grupo Administradores es el propietario
  • El propietario siempre puede cambiar los permisos en un objeto, incluso cuando se le niega todo el acceso al objeto

Toma de posesión

Section titled “Toma de posesión”

La propiedad puede ser tomada por:

  • Cualquier usuario con el derecho de usuario “Tomar posesión de archivos u otros objetos” (por defecto, al grupo Administradores se le otorga este derecho)
  • Cualquier persona o grupo que tenga el permiso “Tomar posesión” en el objeto en cuestión
  • Un usuario que tenga el derecho de usuario “Restaurar archivos y directorios”

Transferencia de propiedad

Section titled “Transferencia de propiedad”

La propiedad se puede transferir de las siguientes maneras:

  1. El propietario actual puede otorgar el permiso “Tomar posesión” a otro usuario, permitiendo que ese usuario tome posesión en cualquier momento
  2. Un administrador puede tomar posesión
  3. Un usuario que tenga el derecho de usuario “Restaurar archivos y directorios” puede asignar la propiedad a cualquier usuario o grupo

Permisos explícitos vs. heredados

Section titled “Permisos explícitos vs. heredados”

Existen dos tipos de permisos:

Permisos explícitos: Son aquellos que se establecen por defecto cuando se crea el objeto o por acción del usuario.

Permisos heredados: Son aquellos que se propagan de un objeto padre a un objeto hijo. Los permisos heredados facilitan la tarea de gestionar permisos y aseguran la coherencia de permisos entre todos los objetos en un contenedor dado.

Por defecto, los objetos que se crean en un contenedor heredan los permisos de ese contenedor cuando se crean los objetos.

Ejemplo: Cuando crea una carpeta llamada MiCarpeta, todas las subcarpetas y archivos que se crean en MiCarpeta heredan automáticamente los permisos de MiCarpeta. MiCarpeta tiene permisos explícitos, mientras que todas las subcarpetas y archivos en ella tienen permisos heredados.

Reglas de precedencia

Section titled “Reglas de precedencia”

Reglas importantes

  • Los permisos Denegar heredados NO impiden el acceso a un objeto si el objeto tiene una entrada de permiso Permitir explícito
  • Los permisos explícitos tienen precedencia sobre los permisos heredados, incluso los permisos Denegar heredados

Estructura de DACL

Section titled “Estructura de DACL”

La Lista de control de acceso discrecional (DACL) de cada carpeta está formada por 2 listas:

  1. DACL heredada (Inherited DACL): Permisos que la carpeta actual hereda de su carpeta padre
  2. DACL explícita (Explicit DACL): Permisos que configura manualmente en la carpeta actual

Permisos en una carpeta compartida

Section titled “Permisos en una carpeta compartida”

El acceso a una carpeta compartida se determina a través de dos conjuntos de entradas de permisos:

  1. Permisos de recurso compartido (Share permissions): Los permisos que se establecen en el recurso compartido
  2. Permisos de archivo y carpeta NTFS (NTFS permissions): Los permisos que se establecen en la carpeta

Relación entre permisos:

  • Los permisos de recurso compartido y los permisos NTFS son independientes entre sí
  • Los permisos de acceso finales en una carpeta compartida se determinan teniendo en cuenta ambos conjuntos
  • Se aplican los permisos más restrictivos

Mejor práctica recomendada

Los administradores experimentados prefieren establecer siempre los permisos de recurso compartido en Control total para Todos y confiar completamente en los permisos NTFS para restringir el acceso.

Recomendaciones finales

  • Asigne derechos de usuario a grupos, no a usuarios individuales
  • Evite el uso de denegaciones explícitas cuando sea posible
  • Use permisos NTFS para controlar el acceso, estableciendo permisos de recurso compartido en “Control total” para “Todos”
  • Documente claramente la estructura de permisos de su organización
  • Revise periódicamente los permisos asignados para mantener la seguridad

Consideraciones de seguridad

  • Los cambios en derechos de usuario pueden tener efectos de largo alcance
  • Solo administradores experimentados deben modificar políticas de derechos
  • Algunos privilegios pueden invalidar permisos establecidos en objetos
  • Los permisos explícitos tienen precedencia sobre los heredados
  • Las denegaciones explícitas pueden crear errores inesperados en la configuración
